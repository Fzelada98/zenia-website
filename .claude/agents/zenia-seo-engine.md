## Daily Workflow

### Step 0: SEO Health triage (BEFORE writing content)

The `seo-health.yml` workflow runs every morning at 06:43 UTC and audits
the live site (top 20 priority URLs + sitemap + robots). Check its last
result before writing.

```bash
RUN=$(gh run list --workflow=seo-health.yml --limit 1 --json conclusion,databaseId,createdAt)
CONCLUSION=$(echo "$RUN" | node -e "const d=JSON.parse(require('fs').readFileSync(0)); console.log(d[0].conclusion)")
RUN_ID=$(echo "$RUN" | node -e "const d=JSON.parse(require('fs').readFileSync(0)); console.log(d[0].databaseId)")
echo "Last health run: $CONCLUSION ($RUN_ID)"
```

- **`success`** → site is healthy, skip to Step 1.
- **`failure`** → download artifact, inspect `criticalIssues[]`, apply auto-fix below, commit on branch `health-fix/{date}`, open PR. Only then proceed.

```bash
gh run download "$RUN_ID" -n "seo-health-$RUN_ID"
cat health-report.json
```

**Auto-fix playbook:**

| `kind` | Fix |
|---|---|
| `canonical-missing` | Open the HTML file (e.g. `es/crm-{vertical}.html`). Add `<link rel="canonical" href="https://zeniapartners.com/{path}">` to `<head>`. If it is a city-landing template generated by a script, fix the generator in `scripts/` and regenerate. |
| `hreflang-short` | Bilingual pages need 3 alternates: `en`, `es`, `x-default`. Add missing `<link rel="alternate" hreflang="...">` tags in `<head>`. |
| `http` (non-200 on priority URL) | Check GitHub Pages deploy. If a recent commit moved/deleted the page, restore it or update the priority list in `.github/scripts/seo-health-check.js`. Do NOT silence the alert unless the page is intentionally gone. |
| `sitemap` | Inspect `sitemap.xml` or the generator that builds it. Common causes: malformed XML, missing `<urlset>` wrapper, URL without `<loc>`. |

If the issue `kind` is not listed above, do NOT guess. Open a GitHub
issue titled `[SEO Health] Unknown pattern: {kind}` with the full
`criticalIssues` entry and skip auto-fix. Then proceed to content.

**6.3 Create PR with auto-merge (capture PR URL to avoid race condition):**
```bash
# Create PR and capture its URL
PR_URL=$(gh pr create \
  --title "blog: {es-slug} + EN" \
  --body "Automated bilingual SEO post by zenia-seo-engine. ES + EN versions with cross-hreflang. Auto-merges after Vercel check." \
  --base main \
  --head HEAD)

echo "PR created: $PR_URL"

# Buffer so GitHub registers the PR before enabling auto-merge
sleep 3

# Enable auto-merge on the specific PR (explicit, not ambiguous)
gh pr merge "$PR_URL" --auto --squash --delete-branch

# Verify auto-merge queued
gh pr view "$PR_URL" --json autoMergeRequest | grep -q "enabledAt" && echo "Auto-merge enabled OK" || echo "WARNING: auto-merge NOT enabled"
```

**If auto-merge fails:** the PR stays open. Fabrizzio gets Vercel email and can merge manually via GitHub UI.

The `--auto` flag queues auto-merge. When GitHub checks pass, PR merges to main automatically. This triggers:
- Email to Fabrizzio (PR created + PR merged)
- GitHub Pages deploy
- GitHub Actions: IndexNow ping, GSC API request indexing, sitemap ping, internal linking, Post for Me social publishing

**DO NOT** push to main directly. The PR flow is REQUIRED for:
- Email notifications
- Audit trail
- Ability to revert with 1 click
- Triggering the full GitHub Actions pipeline
